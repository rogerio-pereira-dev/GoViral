# ADR-018: Cloudflare Turnstile Captcha Without Rate Limit for Users

## Status

Approved

## Context

The data collection form (landing → checkout) is public and a target for bots and abuse. Automated submissions must be reduced without harming real users. Rate limiting by IP or identifier could block legitimate buyers (multiple devices, shared networks, VPNs) and negatively impact revenue. Alternatives include: captcha only; captcha + aggressive rate limit; rate limit only.

## Decision

1. **Adopt Cloudflare Turnstile** as the captcha mechanism on the form. The token generated by Turnstile will be sent on submit and validated on the backend with the Turnstile API before creating the record in `analysis_requests` and the Stripe Checkout session. Submissions without a valid token or with an invalid token are rejected.

2. **Do not apply rate limiting for real users** on form and payment flow endpoints (checkout session creation, etc.). The captcha already acts as a barrier for bots; limiting requests by IP or other identifier could block legitimate buyers and reduce conversion/revenue.

3. **Keep** Stripe webhook validation (ADR-016) and other security measures (ADR-017), except rate limit on user-facing routes in the purchase flow.

## Consequences

- **Positive:** Fewer bot submissions with minimal impact on user experience; no risk of blocking sales due to request limits; Turnstile is lightweight and often invisible to real users in the right mode.
- **Negative:** Manual (human) abuse is not mitigated by rate limit; dependency on Cloudflare Turnstile and validation API availability.
- **Neutral:** In future human-abuse scenarios, selective rate limit (e.g. by email or after N purchases) can be evaluated without changing this decision for the overall flow.
